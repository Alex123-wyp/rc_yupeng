# AI 使用说明

本作业允许使用 AI 工具。我使用 AI 辅助完成了需求拆解、方案比较、代码骨架生成、测试补充和文档整理，但关键工程取舍由我根据题目目标和 MVP 边界确定。

## AI 提供帮助的地方

- 帮助提取 PDF 作业要求，确认交付内容包括 README/设计文档、代码实现和 AI 使用说明。
- 帮助梳理系统边界：哪些问题由通知系统解决，哪些问题不适合放入第一版。
- 帮助比较 RabbitMQ、Kafka、Redis Streams、数据库队列表等方案。
- 帮助生成 Spring Boot 项目结构、JPA entity、REST API、RabbitMQ consumer、outbox publisher 和测试用例。
- 帮助检查第一轮编译错误，并根据实际 Maven 测试结果修正 Spring Boot API 使用方式。
- 帮助整理 README，让设计说明覆盖可靠性、失败处理、取舍和未来演进。

## AI 建议过但没有采纳的内容

- 没有采纳“直接做完整供应商模板配置系统”的建议。题目更关注可靠投递和工程判断，模板系统会扩大第一版范围。
- 没有采纳“使用 Kafka 作为第一版 MQ”的建议。Kafka 适合事件流和高吞吐日志，但本题核心是任务投递、ACK 和重试，RabbitMQ 更直接。
- 没有采纳“追求 exactly-once 投递”的建议。外部 HTTP API 无法由本系统单方面保证 exactly-once，强行承诺会误导设计。
- 没有采纳“加入完整监控、告警、管理后台、多租户、权限系统”的建议。这些是生产演进方向，但不适合作为 MVP 的必要范围。
- 没有采纳“只用内存队列快速实现”的建议。内存队列在服务重启时会丢任务，不符合题目中的可靠性要求。

## 我自己做出的关键决策

### 1. 第一版使用 RabbitMQ

我选择 RabbitMQ 是因为该场景更像“可靠任务队列”而不是“事件流平台”。RabbitMQ 的 durable queue、ACK、consumer worker 模型很适合表达外部 HTTP 通知投递。Kafka 可以作为未来高吞吐事件平台的一部分，但第一版使用 Kafka 会增加重试、状态查询和本地运行复杂度。

### 2. 保留 PostgreSQL 状态表和 outbox

我没有让 RabbitMQ 成为唯一数据源，因为业务上需要查询通知状态、失败原因、重试次数和人工排查信息。PostgreSQL 保存业务状态，RabbitMQ 负责异步解耦。

Outbox 是为了处理一个真实故障点：API 请求已经成功写入数据库，但服务在发布 MQ 前宕机。如果没有 outbox，这条通知会永久丢失；有 outbox 后，后台 publisher 可以继续扫描并恢复发布。

### 3. 投递语义选择 at-least-once

外部 HTTP 调用存在很多无法准确判断的情况，例如供应商已处理但响应超时、本服务更新状态前宕机、网络中断等。因此我选择明确说明 at-least-once，并通过 `Idempotency-Key` 支持供应商侧幂等。

这个选择比承诺 exactly-once 更诚实，也更符合真实工程环境。

### 4. 只按 HTTP status 判断成功失败

题目说明业务系统不关心外部 API 返回值，只需尽量可靠送达。因此 MVP 不解析供应商业务 body。第一版只按 HTTP status 做判断：

- `2xx` 成功
- `429/5xx/超时/网络错误` 重试
- 普通 `4xx` 不重试

如果未来供应商协议更复杂，可以为不同 vendor 增加响应解析器。

### 5. 控制 MVP 范围

我刻意没有把第一版做成完整生产平台。原因是这份作业重点是工程判断，而不是功能数量。一个清晰、可运行、边界明确的系统，比一个包含很多未验证模块的“大而全”方案更适合面试交付。
